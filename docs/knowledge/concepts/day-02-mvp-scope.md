# Day 02：MVP 范围

## 一句话定义

MVP 是能验证核心价值的最小产品范围；本项目的 MVP 是一个只读 Java 后端排障 Agent，它先收集证据，再生成可追溯诊断报告。

## 解决的问题

MVP 解决的是范围失控问题。

如果一开始就同时做 LLM API、MCP、RAG、Memory、多 Agent、审批和自动修复，学习和工程风险都会混在一起，任何失败都难以定位。

Day 02 把第一阶段收敛到：

- 一个用户问题。
- 一组只读证据源。
- 一个固定诊断报告格式。
- 一条明确的禁止执行边界。

## 不解决的问题

MVP 不等于最终产品。

它暂时不解决：

- 自动修复。
- 自动部署。
- 写数据库。
- 多项目自动接入。
- 多 Agent 协作。
- 长期记忆。
- 复杂 RAG。

这些能力需要等只读诊断闭环稳定后再加。

## 在本项目中的位置

MVP 范围决定后续所有实现的边界：

- `agent-app` 只做编排和报告生成。
- `mcp-server` 只暴露只读工具。
- `knowledge-base` 只提供本地文档证据。
- `evals` 验证报告格式、证据引用和安全边界。
- `traces` 记录工具调用和观察结果。

## 最小代码证据

Day 02 仍不写业务代码。

今天的最小证据是文档化的输入、边界和输出结构：

- 输入：一个 Java 后端接口 500 或业务异常问题。
- 过程：只读查询代码、配置、Git 历史和本地文档。
- 输出：区分事实、推断、风险和建议的诊断报告。

## 常见误区

- 误区一：MVP 越完整越好。
- 误区二：Agent MVP 必须包含自动执行能力。
- 误区三：只读工具没有安全风险。
- 误区四：诊断报告只要给结论，不需要证据来源。
- 误区五：先把向量库、多 Agent 和长期记忆都搭好再说。

## 自测问题

1. 本项目 MVP 的首个用户问题是什么？
2. 为什么只读工具仍然要做路径、脱敏和范围限制？
3. 哪些能力明确不属于第一版 MVP？
4. 诊断报告为什么要写风险等级？
5. MVP 和最终产品的区别是什么？

## 今日结论

Day 02 的核心是收敛。第一版排障 Agent 不追求自动修复，而是追求只读、安全、可追溯的诊断闭环。
