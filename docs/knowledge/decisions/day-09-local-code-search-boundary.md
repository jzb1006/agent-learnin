# ADR：Day 09 本地代码搜索工具边界

## 背景

Day 08 已定义工具调用契约：`TroubleshootingTool`、`ToolCall`、`ToolResult` 和 `ToolEvidence`。

Day 09 需要实现第一个真实只读工具 `search_code`，让排障 Agent 能按关键词查找目标 Java 项目源码。该工具后续会迁移到 MCP，因此当前实现要保持职责单一、边界清晰、容易测试。

## 决策

在 `agent-app` 的 `tool` 包中新增 `LocalCodeSearchTool`：

- 实现 `TroubleshootingTool`。
- 只接受工具名 `search_code`。
- 只接受 `keyword` 参数。
- `keyword` 必须非空，且最大 128 字符。
- 构造时传入允许根目录 `allowedRoot`。
- 只扫描允许根目录内的 `.java` 普通文件。
- 跳过符号链接。
- 最多返回 5 条 `ToolEvidence`。
- 空结果返回成功 observation，不返回失败。

## 备选方案

### 方案 A：使用 Java NIO 实现最小关键词搜索

只依赖 JDK，递归扫描 `.java` 文件并按行匹配关键词。

### 方案 B：直接调用系统 `rg`

复用成熟命令行工具，搜索性能更好。

### 方案 C：引入全文索引或语义检索

为源码建立索引，支持更复杂的查询和排序。

## 取舍

选择方案 A。

原因：

- KISS：当前只需要教学用最小闭环，JDK NIO 足够。
- YAGNI：不提前引入索引、Embedding 或额外进程依赖。
- DRY：继续复用 Day 08 的统一工具结果对象。
- SOLID：搜索实现只负责本地代码查询，不承担 Agent 推理。

方案 B 性能更好，但会引入命令可用性、转义、跨平台和进程执行边界。方案 C 适合后续 RAG 阶段，不适合 Day 09。

## 后果

正向后果：

- 第一个真实工具可通过单元测试验证。
- 工具边界清晰，便于 Day 13 迁移 MCP。
- 不依赖外部命令，测试稳定。
- 路径、输入和结果数量都可控。

代价：

- 搜索能力只是大小写敏感的简单包含匹配。
- 每次调用都会遍历文件，未做索引缓存。
- 当前只支持 `.java` 文件。
- 空结果只能说明“当前允许范围内未匹配”，不能证明代码不存在相关逻辑。

## 复查条件

出现以下情况时复查该决策：

- 目标项目源码规模导致搜索明显变慢。
- 需要大小写不敏感、正则、多关键词或路径过滤。
- 需要搜索配置、Markdown、日志样本等非 Java 文件。
- Day 13 迁移 MCP 时需要把路径 roots 交给 MCP Server 管理。
- Day 29 做上下文压缩时需要对搜索结果做更细粒度裁剪。
