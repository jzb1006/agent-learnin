# Day 03 决策记录：Prompt 只定义行为契约，不承载业务事实

## 背景

Day 02 已确定第一版 MVP 是只读 Java 后端排障 Agent。Day 03 需要为 Agent 设计基础指令和诊断报告格式约束。

排障 Agent 很容易走向两个极端：

- 把 Prompt 写得很泛，只说“你是 Java 专家”，导致回答不可控。
- 把大量业务知识、故障规则和项目细节写死在 Prompt，导致不可维护、不可验证。

## 决策

第一版 Prompt 只定义行为契约：

- Agent 身份。
- 只读边界。
- 证据优先原则。
- 工具使用规则。
- 输出格式约束。
- 越权请求处理方式。

Prompt 不承载业务事实：

- 不写死接口字段含义。
- 不写死具体模块结构。
- 不写死历史故障原因。
- 不写死日志错误码解释。

业务事实后续通过工具、MCP Resources、RAG 知识库和用户输入提供。

## 备选方案

### 方案 A：Prompt 只做行为契约

优点：

- 边界清晰。
- 便于测试和评测。
- 后续容易接入 Tool Calling、MCP 和 RAG。
- 不会让 Prompt 变成不可维护的业务知识仓库。

缺点：

- 早期看起来能力较弱，需要工具和知识库逐步补齐。

### 方案 B：把常见排障规则写进 Prompt

优点：

- 短期可以更快得到“像专家”的回答。
- 不需要一开始就准备知识库。

缺点：

- 容易过时。
- 不可追溯。
- 难以判断结论来自证据还是模型先验。
- 后续迁移到 RAG 和 MCP 时容易重复。

### 方案 C：只靠用户每次输入完整上下文

优点：

- Prompt 本身很简单。
- 不需要管理知识库。

缺点：

- 对用户负担过高。
- 每次上下文质量不稳定。
- 无法形成可复用的排障能力。

## 结论

采用方案 A：Prompt 只做行为契约。

## 影响

- Day 04 初始化项目结构时，需要为 Prompt、报告、工具和知识库预留清晰边界。
- Day 07 做结构化输出时，可以沿用 Day 03 的诊断报告结构。
- Day 08-10 设计工具时，每个工具都必须补齐 tool instruction、权限和失败语义。
- Day 21 以后做 RAG 时，业务事实进入知识库，而不是回填到 Prompt。

## 验证方式

后续至少用三类测试验证 Prompt 边界：

- 格式测试：输出是否符合诊断报告结构。
- 证据测试：无证据结论是否标记为推断。
- 安全测试：用户要求重启、部署、改配置、写数据库时，Agent 是否拒绝直接执行并要求人工确认。
