# Day 02 决策记录：第一版只做只读诊断 MVP

## 背景

本项目目标是构建一个 MCP 优先的 Java 后端排障 Agent。为了避免第一阶段范围失控，需要先明确 MVP 的输入、证据源、输出和禁止动作。

## 决策

第一版 MVP 只覆盖一个核心场景：

```text
Java 后端接口出现 500 或业务异常，Agent 基于本地代码、配置、Git 历史和本地文档做只读排查，并输出诊断报告。
```

第一版只允许：

- 查代码。
- 查配置并脱敏。
- 查 Git 历史。
- 查本地文档。
- 生成诊断报告。

第一版禁止默认执行：

- 自动修复代码。
- 修改配置。
- 重启服务。
- 部署。
- 写数据库。
- 调生产 API。
- 删除文件或批量移动文件。

## 备选方案

### 方案 A：只读诊断 MVP

优点：

- 范围清晰，容易验证。
- 安全边界明确。
- 后续可以自然迁移到 MCP Tool。
- 失败时容易定位是计划、工具、证据还是报告问题。

缺点：

- 不能直接修复问题。
- 对用户来说第一版更像排查助手，而不是自动运维系统。

### 方案 B：诊断加自动修复

优点：

- 看起来更接近完整 Agent。
- 用户能直接看到代码变更。

缺点：

- 权限、回滚、测试、责任边界都会提前复杂化。
- 容易把学习重点从 Agent 工程转成普通代码生成。
- 不适合作为第一阶段。

### 方案 C：一开始接入 RAG、Memory 和多 Agent

优点：

- 能覆盖更多 Agent 概念。
- 架构看起来更完整。

缺点：

- 过早引入复杂依赖。
- 每个模块都还没稳定时，整体效果无法评估。
- 不符合 KISS 和 YAGNI。

## 结论

采用方案 A：只读诊断 MVP。

## 影响

- Day 03 的 Prompt 设计要围绕只读诊断报告，而不是自动修复。
- Day 04 的项目结构要让 Agent 编排和工具实现分离。
- Day 09、Day 10 的工具必须先实现路径限制、脱敏和统一结果结构。
- 后续高风险操作只能作为 Human-in-the-loop 设计，不进入第一阶段默认执行链路。

## 验证方式

后续实现完成后，至少用一个接口 500 示例问题验证：

- Agent 是否只调用只读工具。
- 报告是否列出证据来源。
- 推断是否和事实分开。
- 是否拒绝默认执行重启、部署、改配置、写数据库等动作。
