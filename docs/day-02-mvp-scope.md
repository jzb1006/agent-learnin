# Day 02：明确排障 Agent 的 MVP 范围

## 今日节点

Day 02：明确排障 Agent 的 MVP 范围。

## 今日主题

把“Java 后端排障 Agent”从一个大目标收敛成第一阶段可验证、只读、安全的最小闭环。

## 今天解决的问题

- 第一版 MVP 到底接什么输入。
- Agent 第一阶段能查哪些证据。
- Agent 第一阶段不能做哪些高风险动作。
- 首个用户问题如何定义。
- 诊断报告应该长什么样。

## 今天不解决的问题

- 不接入真实 LLM API。
- 不实现 Tool Calling。
- 不搭建 MCP Server。
- 不做 RAG 检索实现。
- 不创建 Spring Boot 工程。
- 不执行重启、部署、改配置、写数据库、调用生产 API 等高风险动作。

## 一句话定义 MVP

本项目 Day 02 定义的 MVP 是：用户提交一个 Java 后端故障问题后，Agent 只通过本地只读证据查询，生成一份区分事实、推断、风险和建议的诊断报告。

这里的重点不是“能自动修好问题”，而是“能按边界收集证据并给出可追溯诊断”。

## MVP 用户画像

第一版服务对象是 Java 后端开发者。

典型状态：

- 已经知道某个接口、任务或功能异常。
- 不确定问题来自代码、配置、最近提交还是文档理解偏差。
- 希望 Agent 先帮忙收集证据、整理可能原因和下一步检查建议。
- 不希望 Agent 自动改代码、改配置、重启服务或操作数据库。

## 首个用户问题

第一版只覆盖一个问题模板：

```text
某个 Java 后端接口出现 500 或业务异常，请帮我基于当前项目代码、配置、Git 历史和本地文档做只读排查，并输出诊断报告。
```

示例输入：

```text
用户登录接口最近返回 500，请帮我排查可能原因。
```

这个问题适合作为 MVP，因为它天然需要多源证据：

- 代码：接口入口、Service 调用链、异常处理。
- 配置：profile、外部依赖、开关、超时配置。
- Git 历史：最近是否改过相关模块。
- 文档：接口约定、字段含义、已知排障手册。

## 只读接入范围

MVP 只允许接入本地、只读、可追溯的证据源。

| 能力 | 第一版目标 | 只读边界 |
| --- | --- | --- |
| `search_code` | 按关键词搜索目标 Java 项目源码 | 只能读取允许根目录内文件 |
| `read_config` | 读取配置文件并脱敏 | 不写配置，不展示敏感值 |
| `git_history` | 查询最近提交和相关文件变更 | 不切分支，不 reset，不提交 |
| `retrieve_docs` | 查询本地接口文档、排障手册、历史案例 | 不访问生产系统，不调用外部 API |
| `generate_report` | 汇总证据并生成诊断报告 | 报告必须标注事实和推断 |

第一阶段可以把这些能力先当作接口设计，后续 Day 再逐步实现。

## 明确排除范围

MVP 不包含以下能力：

- 自动修复代码。
- 自动修改配置。
- 自动重启服务。
- 自动部署。
- 自动写数据库或执行 SQL。
- 自动调用生产 API。
- 自动创建 Jira、TAPD 或工单。
- 多 Agent 协作。
- 长期记忆写入。
- 向量数据库和复杂 RAG。
- 自动选择多个真实项目。

这些能力不是永远不能做，而是不属于第一版 MVP。先把只读诊断闭环做稳，再讨论审批式写操作。

## 诊断报告格式

第一版报告使用固定结构，便于后续做结构化输出和测试。

```markdown
# 诊断报告

## 1. 问题摘要

- 用户问题：
- 影响对象：
- 当前结论：
- 风险等级：low / medium / high / unknown

## 2. 已检查证据

| 证据类型 | 来源 | 关键发现 | 可信度 |
| --- | --- | --- | --- |
| code | 文件路径和行号 | 发现了什么 | high / medium / low |
| config | 配置路径 | 发现了什么 | high / medium / low |
| git | commit 或文件历史 | 发现了什么 | high / medium / low |
| docs | 文档路径 | 发现了什么 | high / medium / low |

## 3. 可能原因

1. 原因 A
   - 支撑证据：
   - 反证或不确定点：
2. 原因 B
   - 支撑证据：
   - 反证或不确定点：

## 4. 建议的下一步

- 只读验证：
- 需要人工确认的操作：
- 暂不建议执行的操作：

## 5. 边界声明

- 本报告基于只读证据生成。
- 未执行重启、部署、配置修改、数据库写入或生产 API 调用。
- 没有证据支撑的结论标记为推断。
```

## 项目映射

Day 02 仍然不写业务代码，但已经为后续目录职责定边界。

```text
projects/mcp-troubleshooting-agent/
  agent-app/        # 后续负责 Agent 编排、报告生成、结构化输出
  mcp-server/       # 后续负责 search_code/read_config/git_history 等只读工具
  knowledge-base/   # 后续放接口文档、排障手册和历史案例
  evals/            # 后续放报告格式、证据引用和安全边界评测样例
  traces/           # 后续记录每次工具调用链路和观察结果
```

## KISS / YAGNI / DRY / SOLID 落地

KISS：第一版只处理一个排障入口，不做所有故障类型。

YAGNI：暂不引入多 Agent、向量库、长期记忆、自动修复和审批工作流。

DRY：后续所有工具结果统一进入诊断报告结构，不为每个工具单独设计一套报告格式。

SOLID：Agent 负责编排和判断，工具负责只读取证，报告负责表达结论；三者职责分开。

## 验收标准

完成 Day 02 时，你应该能解释：

- MVP 为什么只选一个用户问题。
- 为什么第一阶段必须坚持只读边界。
- 为什么诊断报告要区分事实、推断、风险和建议。
- 哪些能力被明确排除在第一版之外。
- 后续实现时各目录分别承担什么职责。

## 复习问题

1. 为什么 MVP 不应该一开始就覆盖所有排障场景？
2. `search_code` 和 `read_config` 都是只读工具，它们的风险边界分别是什么？
3. 为什么诊断报告里要写“反证或不确定点”？
4. 如果用户要求“顺便帮我重启服务试试”，第一阶段 Agent 应该怎么处理？
5. 第一版为什么不引入向量数据库和长期记忆？
