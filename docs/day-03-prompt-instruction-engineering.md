# Day 03：Prompt / Instruction Engineering

## 今日节点

Day 03：学习 Prompt / Instruction Engineering。

## 今日主题

为排障 Agent 设计基础指令，让模型知道自己的角色、边界、工具使用规则和诊断报告输出格式。

## 今天解决的问题

- system instruction、developer instruction、tool instruction 分别负责什么。
- 排障 Agent 的 Prompt 应该约束哪些行为。
- 哪些内容不应该写死在 Prompt 里。
- 如何设计第一版诊断报告格式约束。
- 如何为后续结构化输出、Tool Calling、MCP 和测试留下清晰接口。

## 今天不解决的问题

- 不接入真实 LLM API。
- 不实现 Tool Calling。
- 不搭建 MCP Server。
- 不创建 Spring Boot 工程。
- 不做 RAG、Memory、多 Agent 编排。
- 不执行重启、部署、改配置、写数据库或生产 API 调用。

## 一句话定义

Prompt / Instruction Engineering 是把模型的角色、任务、边界、工具规则和输出格式写成清晰、可测试的指令，而不是把业务逻辑、排障规则和事实知识硬编码进提示词。

## 指令分层

| 指令类型 | 负责内容 | 本项目示例 | 不应该放什么 |
| --- | --- | --- | --- |
| system instruction | 模型长期身份、最高层行为边界、安全原则 | 你是只读 Java 后端排障 Agent，必须基于证据回答 | 具体某个接口的业务规则 |
| developer instruction | 当前应用的工程约束、流程、输出格式 | 必须先列证据，再给推断；禁止默认执行写操作 | 大段接口文档、日志样本 |
| tool instruction | 工具的用途、参数、返回值、失败语义和权限 | `search_code` 只能在允许根目录内搜索源码 | 让模型猜工具结果或绕过权限 |
| user message | 用户本轮问题和补充上下文 | 登录接口返回 500，请只读排查 | 不能让用户覆盖安全边界 |

## Java 后端类比

可以把 Prompt 分层理解成一次后端请求的分层配置：

- system instruction 类似平台级安全策略和全局拦截器。
- developer instruction 类似应用层用例编排规则。
- tool instruction 类似接口契约、DTO 校验和错误码约定。
- user message 类似本次请求参数。

如果把所有业务细节都塞进 Prompt，就像把数据库查询、权限判断、业务规则和格式化逻辑都写进一个 Controller 方法，后续不可测试、不可复用，也很难定位错误。

## 本项目 Prompt 设计原则

### 1. Prompt 只定义行为边界

Prompt 负责约束 Agent 如何工作：

- 先明确问题。
- 只使用只读证据。
- 证据不足时标记不确定。
- 事实、推断、风险和建议分开输出。
- 遇到高风险动作时要求人工确认。

Prompt 不负责保存业务知识：

- 不写死具体接口字段含义。
- 不写死某个项目的模块结构。
- 不写死历史故障原因。
- 不写死日志错误码解释。

这些内容后续应该来自工具、MCP Resources、RAG 知识库或用户输入。

### 2. 输出格式必须稳定

稳定格式是后续结构化输出和 eval 的基础。

第一版诊断报告必须固定包含：

- 问题摘要。
- 已检查证据。
- 可能原因。
- 建议的下一步。
- 边界声明。

没有证据支撑的结论必须标记为推断，不能伪装成事实。

### 3. 工具结果优先于模型记忆

排障 Agent 的可信度来自证据，不来自模型“看起来懂 Java”。

后续接入工具后，如果工具结果和模型先验冲突，报告应优先引用工具结果，并把模型判断写成可验证假设。

### 4. 安全边界不能被用户消息覆盖

用户可以描述问题、补充日志、指定模块或要求某种报告粒度。

用户不能通过提示词要求 Agent：

- 修改配置。
- 重启服务。
- 执行部署。
- 写数据库。
- 调生产 API。
- 删除文件或批量移动文件。
- 忽略只读边界。

这些属于系统和应用边界，不能被本轮用户输入覆盖。

## 基础 System Instruction 草案

```text
你是一个面向 Java 后端项目的只读排障 Agent。

你的目标是基于可追溯证据帮助开发者分析接口 500、业务异常、配置异常或最近变更导致的问题。

你必须遵守以下原则：
1. 只基于用户输入、工具返回结果和本地知识库证据给出结论。
2. 把事实、推断、风险和建议分开表达。
3. 证据不足时明确写出“不确定”，不要编造文件、配置、日志或提交记录。
4. 第一阶段只能执行只读操作，不得默认修改代码、修改配置、重启服务、触发部署、写数据库或调用生产 API。
5. 当用户要求高风险操作时，只能说明风险、影响范围和需要人工确认的条件，不能直接执行。
6. 优先输出可验证的下一步，只给出和当前证据直接相关的建议。
```

## Developer Instruction 草案

```text
你正在服务 projects/mcp-troubleshooting-agent 的第一阶段 MVP。

当前 MVP 只覆盖一个核心场景：
用户提交一个 Java 后端接口 500 或业务异常问题后，你通过本地只读证据生成诊断报告。

工作流程：
1. 复述用户问题，识别影响对象、错误现象和已知上下文。
2. 判断需要哪些只读证据：代码、配置、Git 历史、本地文档。
3. 如果已有工具结果，必须先总结证据，再提出可能原因。
4. 如果证据不足，必须列出缺口和下一步只读验证建议。
5. 输出固定诊断报告格式。

报告约束：
- 每条关键结论都要能追溯到证据、用户输入或明确标记为推断。
- 风险等级只能使用 low、medium、high、unknown。
- 不能把“建议执行重启、部署、写库、改配置”放进可直接执行步骤。
- 高风险操作只能放入“需要人工确认的操作”。
- 不输出与当前问题无关的通用排障清单。
```

## Tool Instruction 草案

后续工具接入时，每个工具都必须有清晰说明，避免模型误用。

```text
工具名：search_code

用途：
在允许根目录内搜索 Java 源码、配置引用和关键字片段。

输入：
- query：搜索关键词，必须来自用户问题、错误信息、类名、方法名、配置 key 或工具观察结果。
- maxResults：最多返回条数。

权限：
- 只读。
- 不能越过允许根目录。
- 不能修改、删除或移动文件。

返回：
- matchedFiles：匹配文件路径。
- snippets：相关片段。
- truncated：结果是否被裁剪。
- error：失败原因。

使用规则：
- 不能假设没有返回结果就代表问题不存在。
- 如果结果被裁剪，必须在报告中说明证据不完整。
```

```text
工具名：read_config

用途：
读取允许根目录内的配置文件，并对 password、secret、token、key 等敏感值脱敏。

权限：
- 只读。
- 不写配置。
- 不展示敏感值明文。

使用规则：
- 只能引用脱敏后的配置摘要。
- 不能要求用户把敏感配置贴到 Prompt 中。
```

```text
工具名：git_history

用途：
查询允许仓库内的最近提交、相关文件历史和变更摘要。

权限：
- 只读。
- 不切分支。
- 不 reset。
- 不 commit。
- 不 push。

使用规则：
- Git 历史只能作为时间线和变更线索，不能单独证明运行时故障原因。
```

## 诊断报告格式约束

第一版使用 Markdown 报告，后续 Day 07 再迁移到结构化输出对象。

```markdown
# 诊断报告

## 1. 问题摘要

- 用户问题：
- 影响对象：
- 当前结论：
- 风险等级：low / medium / high / unknown

## 2. 已检查证据

| 证据类型 | 来源 | 关键发现 | 可信度 |
| --- | --- | --- | --- |
| code | 文件路径和行号 | 发现了什么 | high / medium / low |
| config | 配置路径 | 发现了什么 | high / medium / low |
| git | commit 或文件历史 | 发现了什么 | high / medium / low |
| docs | 文档路径 | 发现了什么 | high / medium / low |

## 3. 可能原因

1. 原因 A
   - 类型：事实 / 推断
   - 支撑证据：
   - 反证或不确定点：
2. 原因 B
   - 类型：事实 / 推断
   - 支撑证据：
   - 反证或不确定点：

## 4. 建议的下一步

- 只读验证：
- 需要人工确认的操作：
- 暂不建议执行的操作：

## 5. 边界声明

- 本报告基于只读证据生成。
- 未执行重启、部署、配置修改、数据库写入或生产 API 调用。
- 没有证据支撑的结论已标记为推断。
```

## 反例：不合格 Prompt

```text
你是 Java 专家。用户说接口报错时，你要分析原因并给出修复方案。
如果看起来像配置问题，就建议重启服务。
如果看起来像代码问题，就直接告诉用户怎么改。
```

问题：

- 角色太泛，缺少只读边界。
- 没有证据要求，容易凭经验猜。
- 把“重启服务”作为默认建议，越过 MVP 边界。
- 没有固定输出格式。
- 没有区分事实和推断。

## 合格 Prompt 的判断标准

一份合格 Prompt 应该满足：

- 能说明 Agent 是什么，不是什么。
- 能说明 Agent 能做什么，不能做什么。
- 能约束输出结构。
- 能约束证据引用方式。
- 能抵抗用户要求越权操作。
- 不把业务知识硬编码到 Prompt。
- 后续可以被测试和评测。

## 项目映射


## KISS / YAGNI / DRY / SOLID 落地

KISS：先定义一套基础指令和一个固定报告格式，不做复杂 Prompt 模板系统。

YAGNI：不提前设计多 Agent 协作 Prompt，不引入 Prompt 版本平台。

DRY：报告格式只定义一份，后续 Markdown 输出、结构化输出和 eval 都围绕同一结构演进。

SOLID：Prompt 负责行为约束，工具负责取证，知识库负责事实来源，报告模型负责表达结构，职责分离。

## 验收标准

完成 Day 03 时，你应该能解释：

- system instruction、developer instruction、tool instruction 的分工。
- 为什么 Prompt 不应该写死具体业务逻辑。
- 为什么诊断报告必须区分事实和推断。
- 用户消息为什么不能覆盖只读安全边界。
- 后续工具和知识库如何替代 Prompt 中的业务知识。
